# libdnet 跨平台底层实现源码分析

## 架构设计概述

libdnet 采用统一的抽象接口层（API），底层针对不同操作系统实现具体的功能模块。通过编译时检测系统特性，自动选择对应的实现文件。

## 平台检测机制

在 `configure.ac` 中通过以下机制选择实现：

```c
// 219-228行: ARP 接口检测
if test "$ac_cv_header_iphlpapi_h" = yes ; then
    AC_LIBOBJ([arp-win32])      // Windows
elif test "$ac_cv_dnet_ioctl_arp" = yes ; then
    AC_LIBOBJ([arp-ioctl])     // 通用 ioctl
elif test "$ac_cv_dnet_route_h_has_rt_msghdr" = yes ; then
    AC_LIBOBJ([arp-bsd])       // BSD/macOS
else
    AC_LIBOBJ([arp-none])       // 不支持的平台
```

## 各模块平台实现对比

### 1. 以太网接口

#### Linux 实现 (`eth-linux.c`)

**核心机制：PF_PACKET 套接字**

```c
// 42-67行: 打开以太网设备
eth_t *eth_open(const char *device)
{
    // 创建 PF_PACKET 原始套接字
    e->fd = socket(PF_PACKET, SOCK_RAW, htons(ETH_P_ALL));
    
    // 设置广播选项
    setsockopt(e->fd, SOL_SOCKET, SO_BROADCAST, &n, sizeof(n));
    
    // 获取接口索引
    ioctl(e->fd, SIOCGIFINDEX, &e->ifr);
    
    // 设置链路层地址结构
    e->sll.sll_family = AF_PACKET;
    e->sll.sll_ifindex = e->ifr.ifr_ifindex;
}

// 69-78行: 发送以太网帧
ssize_t eth_send(eth_t *e, const void *buf, size_t len)
{
    struct eth_hdr *eth = (struct eth_hdr *)buf;
    e->sll.sll_protocol = eth->eth_type;
    return sendto(e->fd, buf, len, 0, 
                  (struct sockaddr *)&e->sll, sizeof(e->sll));
}
```

**关键特性：**
- 使用 `PF_PACKET` 协议族直接访问数据链路层
- `sockaddr_ll` 结构指定物理接口
- 通过 `ioctl` 获取/设置硬件地址

---

#### BSD/macOS 实现 (`eth-bsd.c`)

**核心机制：BPF (Berkeley Packet Filter)**

```c
// 39-74行: 打开 BPF 设备
eth_t *eth_open(const char *device)
{
    // 尝试打开 /dev/bpf0-127
    for (i = 0; i < 128; i++) {
        snprintf(file, sizeof(file), "/dev/bpf%d", i);
        e->fd = open(file, O_RDWR);
        if (e->fd != -1 || errno != EBUSY)
            break;
    }
    
    // 绑定到指定网络接口
    ioctl(e->fd, BIOCSETIF, (char *)&ifr);
    
    // 设置内核补全头部（不自动填充源 MAC）
    ioctl(e->fd, BIOCSHDRCMPLT, &i);
}

// 76-80行: 发送数据
ssize_t eth_send(eth_t *e, const void *buf, size_t len)
{
    return write(e->fd, buf, len);  // 直接写入 BPF 设备
}

// 94-138行: 获取 MAC 地址
int eth_get(eth_t *e, eth_addr_t *ea)
{
    // 通过 sysctl 获取路由表信息
    int mib[] = { CTL_NET, AF_ROUTE, 0, AF_LINK, NET_RT_IFLIST, 0 };
    sysctl(mib, 6, buf, &len, NULL, 0);
    
    // 解析 if_msghdr 和 sockaddr_dl
    ifm = (struct if_msghdr *)p;
    sdl = (struct sockaddr_dl *)(ifm + 1);
}
```

**关键特性：**
- 使用 `/dev/bpf*` 设备文件
- 通过 `BIOCSETIF` 绑定到网络接口
- 使用 `sysctl` 系统调用获取接口信息

---

#### Windows 实现 (`eth-win32.c`)

**现状：Windows 原生实现为空**

```c
// 所有函数都返回错误
eth_t *eth_open(const char *device) { return (NULL); }
ssize_t eth_send(eth_t *eth, const void *buf, size_t len) { return (-1); }
```

**实际使用：**
- 依赖 WinPcap 库提供底层能力
- WinPcap 提供了 `pcap_sendpacket()` 等函数

### 3. 路由接口

#### Linux 实现 (`route-linux.c`)

**核心机制：Netlink + Procfs**

```c
// 46-69行: 打开 Netlink 套接字
route_t *route_open(void)
{
    // 普通路由操作使用 AF_INET
    r->fd = socket(AF_INET, SOCK_DGRAM, 0);
    
    // 查询使用 Netlink
    r->nlfd = socket(AF_NETLINK, SOCK_RAW, NETLINK_ROUTE);
    bind(r->nlfd, (struct sockaddr *)&snl, sizeof(snl));
}

// 71-92行: 添加路由
int route_add(route_t *r, const struct route_entry *entry)
{
    // 使用传统的 ioctl 方式
    ioctl(r->fd, SIOCADDRT, &rt);
}

// 117-221行: 查询路由
int route_get(route_t *r, struct route_entry *entry)
{
    // 构造 Netlink 消息
    nmsg->nlmsg_type = RTM_GETROUTE;
    
    // 发送请求
    sendmsg(r->nlfd, &msg, 0);
    
    // 接收响应
    recvmsg(r->nlfd, &msg, 0);
}

// 224-297行: 遍历路由表
int route_loop(route_t *r, route_handler callback, void *arg)
{
    // IPv4: 读取 /proc/net/route
    fp = fopen("/proc/net/route", "r");
    
    // IPv6: 读取 /proc/net/ipv6_route
    fp = fopen("/proc/net/ipv6_route", "r");
}
```

**关键特性：**
- 混合使用 Netlink（查询）和 ioctl（修改）
- 读取 `/proc` 文件系统获取路由表
- Netlink 提供更丰富的路由信息

### 4. 防火墙接口

#### Linux IPTables 实现 (`fw-ipchains.c`)

**核心机制：setsockopt**

```c
// 114-124行: 打开防火墙接口
fw_t *fw_open(void)
{
    fw->fd = socket(AF_INET, SOCK_RAW, IPPROTO_RAW);
}

// 127-135行: 添加规则
int fw_add(fw_t *fw, const struct fw_rule *rule)
{
    // 将规则转换为内核格式
    fr_to_fwc(rule, &fwc);
    
    // 通过 setsockopt 添加
    setsockopt(fw->fd, IPPROTO_IP, IP_FW_APPEND, &fwc, sizeof(fwc));
}

// 149-216行: 遍历规则
int fw_loop(fw_t *fw, fw_handler callback, void *arg)
{
    // 读取 /proc/net/ip_fwchains
    fp = fopen("/proc/net/ip_fwchains", "r");
    
    // 解析每一行规则
    sscanf(buf, "%8s %X/%X->%X/%X ...", ...);
}
```

**注意：**
- 支持 ipchains（老版本）、ipfw（BSD）等
- 读取 `/proc` 获取当前规则

### 5. TUN/TAP 接口

#### Linux 实现 (`tun-linux.c`)

**核心机制：/dev/net/tun**

```c
// 35-64行: 打开 TUN 设备
tun_t *tun_open(struct addr *src, struct addr *dst, int mtu)
{
    // 打开 TUN/TAP 设备
    tun->fd = open("/dev/net/tun", O_RDWR, 0);
    
    // 设置为 TUN 模式（三层）
    tun->ifr.ifr_flags = IFF_TUN;
    ioctl(tun->fd, TUNSETIFF, (void *) &tun->ifr);
    
    // 配置接口参数
    intf_set(tun->intf, &ifent);
}

// 78-90行: 发送数据
ssize_t tun_send(tun_t *tun, const void *buf, size_t size)
{
    struct iovec iov[2];
    uint32_t etype = htonl(ETH_TYPE_IP);
    
    // 写入协议类型 + 数据
    iov[0].iov_base = &etype;
    iov[0].iov_len = sizeof(etype);
    iov[1].iov_base = (void *)buf;
    iov[1].iov_len = size;
    
    return writev(tun->fd, iov, 2);
}

// 92-104行: 接收数据
ssize_t tun_recv(tun_t *tun, void *buf, size_t size)
{
    // 读取协议类型 + 数据
    readv(tun->fd, iov, 2) - sizeof(type);
}
```

**关键特性：**
- `/dev/net/tun` 是字符设备
- `TUNSETIFF` ioctl 配置设备
- 数据包含 4 字节协议头 + 实际 IP 包

## 平台差异总结

| 功能模块 | Linux | BSD/macOS | Windows |
|---------|-------|-----------|---------|
| 以太网 | PF_PACKET + ioctl | BPF + sysctl | WinPcap |
| ARP | ioctl | Route Socket | Win32 API |
| 路由 | Netlink + procfs | Route Socket | Win32 API |
| 防火墙 | setsockopt + procfs | PF ioctl | PktFilter |
| TUN/TAP | /dev/net/tun | /dev/tun* | OpenVPN TAP |
| 接口配置 | ioctl | ioctl + sysctl | IP Helper API |

## 统一抽象层

所有平台实现都遵循相同的 API 接口，定义在 `include/dnet.h`：

```c
// 统一的接口结构
struct eth_handle { ... };    // 平台特定
struct route_handle { ... };  // 平台特定

// 统一的 API 函数
eth_t *eth_open(const char *device);
ssize_t eth_send(eth_t *e, const void *buf, size_t len);

route_t *route_open(void);
int route_add(route_t *r, const struct route_entry *entry);
```

## 设计模式

1. **策略模式**：不同平台选择不同的实现策略
2. **编译时多态**：通过 autoconf 检测系统特性，链接对应的 .o 文件
3. **适配器模式**：将不同的系统 API 适配到统一的接口
